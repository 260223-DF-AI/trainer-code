# P2 - Cloud-Native High-Performance Pipeline

## Application Overview

An upgrade to the Data Ingestion Subsystem (Phase 1) focused on scalability, lowering operation costs, and transitioning to a **Cloud-Native Data Stack** within Google Cloud Platform (GCP).

Phase 2 transitions from local PostgreSQL storage to keeping file structures in **Parquet columnar format**, pushing processed streams into **Google Cloud Storage (GCS)**, and querying via **BigQuery**.

It will:

- Read local file streams in memory-efficient chunks.
- Convert raw files (CSV) into **Parquet** files.
- Load the records into **Google Cloud Storage** as a data lake layer.
- Create or update **BigQuery External Tables** for fast, large-scale analytics with Always-Free tier capabilities.
- Build a **FastAPI Microservice** to orchestrate file cleaning and query execution via HTTP.

### Core Functional Scope

#### As a data engineer, I want to

- Use Python standard libraries (or `Pandas`) to convert files into dense **Parquet partitions**.
- Authenticate to GCP and programmatically upload `.parquet` bundles with checksum guards.
- Formulate schema definitions dynamically so BigQuery can read column groups automatically.
- Maintain a local or cloud-based **Audit Log** preventing duplicate bundle downloads or uploads.

#### As an API client, I want to

- Send an HTTP `POST` request to trigger `.csv` to `.parquet` conversion pipelines.
- Send an HTTP `GET` request specifying metric parameters to trigger a **BigQuery** query and fetch row JSONs safely.

#### Cloud destinations

- **Google Cloud Storage (GCS)** – Serves as the raw/cleaned Data Lake housing parquet files.
- **BigQuery** – Serves as the Data Warehouse querying the GCS storage bundles natively.

#### Standard Functional Scope

1. **Columnar Compressing**: Save cleaned Phase 1 loads in `.parquet` buffers.
2. **Cloud Guard**: Check bucket content matching prior hashes to bypass processing steps if identical files are detected.
3. **Batch Cloud Upgrades**: Employ streaming buffers up to Cloud Storage backends to keep execution RAM footprint capped (e.g., 256MB).
4. **Resilience**: Handle token expirations or request throttles securely with backoffs.
5. **API Endpoints Services**:
   - `POST /convert`: Triggers retrieval of `.csv` from a GCS bucket path and processes it into a `.parquet` cloud asset dynamically.
   - `GET /query`: Parameterizes SQL fetches through BigQuery to return structured JSON payloads to browser devices.

### Definition of Done

The Cloud-Native Pipeline is considered complete when:

- It successfully validates local datasets, writes locally-partitioned `.parquet` nodes, and uploads them to a **GCS bucket**.
- Data can be queried inside **GCP BigQuery** without uploading native database engine overheads.
- Operates fully within the **GCP Always Free Tier** limits (under 5GB GCS, under 10GB BigQuery triggers).
- FastAPI endpoints execution confirms reliable HTTP responses (200 OK) with validation logs safely printed.
- Documentation provided on setting up a Google Service Account (`.json` key management).

### Non-Functional Expectations

- Benchmarking report showing: `Disk Space Savings (%)`, `Upload speed (s)`, `Query Access duration (s)`.
- Use correct folder hierarchies in GCS (e.g., `bucket/stg_sales/year=2026/month=03/` partition structure).

## Agile Operations & Team Management

To simulate scaling in standard enterprise environments, teams will adopt an **Agile project management cycle**:

- **Team Capacity:** 3 members per team.
- **Daily Standups (15 min):** Focused updates answer:
  - *What was accomplished yesterday?*
  - *What is the plan for today?*
  - *Any blocker/impediment restricting development?*
- **Visual Trackers (Kanban Board):** Use tools like Trello, Jira, or GitHub Projects:
  - All work items must be broken into separate atomic sub-tasks.
  - Tracks should maintain statuses like `To-Do`, `In Progress`, `PR / Review`, and `Completed`.
- **Collaborative Work & Regular PRs:** All development is highly collaborative.
  - Work must be combined via **frequent Pull Requests (PRs)** to ensure integration safety.
  - Every PR requires **teammate Code Reviews** and approvals before being merged into the core branch.
  - Pair-programming is highly encouraged for core data transformations and GCP deployments to mitigate solo bugs.
