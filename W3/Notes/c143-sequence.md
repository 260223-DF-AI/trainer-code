# Sequence

## Learning Objectives

- Understand sequences as generators of unique numbers
- Create and manage sequences in PostgreSQL
- Use sequences with SERIAL/IDENTITY and independently
- Apply sequences for custom numbering requirements

## Why This Matters

Sequences are the engine behind auto-generated IDs in PostgreSQL. They guarantee unique, sequential numbers even with concurrent inserts. Understanding sequences helps you design better primary keys, implement custom numbering systems, and troubleshoot auto-increment issues.

## The Concept

### What is a Sequence?

A sequence is a database object that generates a series of unique integers. Unlike regular data, sequences maintain their own state and guarantee uniqueness across transactions.

```
Sequence: order_num_seq
Current Value: 1003

NEXTVAL() -> 1004
NEXTVAL() -> 1005
NEXTVAL() -> 1006
...
```

### Creating Sequences

```sql
-- Basic sequence
CREATE SEQUENCE order_number_seq;

-- Sequence with options
CREATE SEQUENCE invoice_seq
    START WITH 1000
    INCREMENT BY 1
    MINVALUE 1000
    MAXVALUE 9999999
    NO CYCLE;

-- Descending sequence
CREATE SEQUENCE countdown_seq
    START WITH 100
    INCREMENT BY -1
    MINVALUE 1;
```

### Sequence Options

| Option | Description | Default |
|--------|-------------|---------|
| START WITH | Initial value | 1 |
| INCREMENT BY | Step between values | 1 |
| MINVALUE | Lowest allowed value | 1 |
| MAXVALUE | Highest allowed value | 2^63-1 |
| CYCLE / NO CYCLE | Wrap around at limit | NO CYCLE |
| CACHE | Values to pre-allocate | 1 |

### Using Sequences

```sql
-- Get next value (increments the sequence)
SELECT NEXTVAL('order_number_seq');

-- Get current value (without incrementing)
SELECT CURRVAL('order_number_seq');

-- Set to a specific value
SELECT SETVAL('order_number_seq', 2000);

-- Set to value, next call returns that value
SELECT SETVAL('order_number_seq', 2000, false);
```

### Sequences and SERIAL

SERIAL is syntactic sugar that creates a sequence automatically:

```sql
-- This:
CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    name VARCHAR(100)
);

-- Is equivalent to:
CREATE SEQUENCE products_product_id_seq;
CREATE TABLE products (
    product_id INTEGER NOT NULL DEFAULT NEXTVAL('products_product_id_seq'),
    name VARCHAR(100)
);
ALTER SEQUENCE products_product_id_seq OWNED BY products.product_id;
```

### IDENTITY Columns (SQL Standard)

PostgreSQL 10+ supports IDENTITY as a standard alternative to SERIAL:

```sql
-- GENERATED ALWAYS: PostgreSQL always generates the value
CREATE TABLE orders (
    order_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    description TEXT
);

-- GENERATED BY DEFAULT: You can override if needed
CREATE TABLE items (
    item_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name TEXT
);

-- Override identity value
INSERT INTO items (item_id, name) 
OVERRIDING SYSTEM VALUE 
VALUES (999, 'Special Item');
```

### Independent Sequences

Use sequences independently from tables:

```sql
-- Invoice numbering starting from year
CREATE SEQUENCE invoice_2024_seq START WITH 1;

-- Generate invoice numbers
SELECT 'INV-2024-' || LPAD(NEXTVAL('invoice_2024_seq')::TEXT, 5, '0');
-- Result: INV-2024-00001

SELECT 'INV-2024-' || LPAD(NEXTVAL('invoice_2024_seq')::TEXT, 5, '0');
-- Result: INV-2024-00002
```

### Sequence in INSERT

```sql
-- Using sequence in insert
INSERT INTO orders (order_num, customer, amount)
VALUES (NEXTVAL('order_number_seq'), 'Alice', 99.99);

-- Multiple inserts share the sequence
INSERT INTO orders (order_num, customer, amount) VALUES
    (NEXTVAL('order_number_seq'), 'Bob', 50.00),
    (NEXTVAL('order_number_seq'), 'Carol', 75.00);
```

### Managing Sequences

```sql
-- View sequence metadata
SELECT * FROM pg_sequences WHERE sequencename = 'order_number_seq';

-- Alter sequence
ALTER SEQUENCE order_number_seq RESTART WITH 5000;
ALTER SEQUENCE order_number_seq INCREMENT BY 10;

-- Drop sequence
DROP SEQUENCE order_number_seq;
DROP SEQUENCE IF EXISTS order_number_seq;
```

### Gaps in Sequences

Sequences can have gaps - this is normal and expected:

```sql
BEGIN;
    SELECT NEXTVAL('order_seq');  -- Returns 101
    -- Transaction fails or is rolled back
ROLLBACK;

-- Next call still returns 102, not 101
SELECT NEXTVAL('order_seq');  -- Returns 102
```

Gaps occur due to:

- Rolled back transactions
- Cached values not used
- This is by design for performance

### Complete Example

```sql
-- Custom order numbering system
CREATE SEQUENCE order_number_seq
    START WITH 10000
    INCREMENT BY 1;

CREATE TABLE orders (
    order_id SERIAL PRIMARY KEY,
    order_number VARCHAR(20) NOT NULL UNIQUE,
    customer_name VARCHAR(100),
    created_at TIMESTAMP DEFAULT NOW()
);

-- Function to generate order number
CREATE OR REPLACE FUNCTION generate_order_number()
RETURNS VARCHAR(20) AS $$
BEGIN
    RETURN 'ORD-' || TO_CHAR(NOW(), 'YYYY') || '-' || 
           LPAD(NEXTVAL('order_number_seq')::TEXT, 6, '0');
END;
$$ LANGUAGE plpgsql;

-- Insert with generated order number
INSERT INTO orders (order_number, customer_name)
VALUES (generate_order_number(), 'Alice');

INSERT INTO orders (order_number, customer_name)
VALUES (generate_order_number(), 'Bob');

SELECT * FROM orders;
-- order_id | order_number      | customer_name | created_at
-- ---------|-------------------|---------------|--------------------
-- 1        | ORD-2024-010000   | Alice         | 2024-01-15 10:30:00
-- 2        | ORD-2024-010001   | Bob           | 2024-01-15 10:31:00

-- Check sequence state
SELECT CURRVAL('order_number_seq');  -- 10001
```

## Key Takeaways

- Sequences generate unique integers automatically
- SERIAL creates a sequence automatically for a column
- IDENTITY is the SQL standard alternative to SERIAL
- Use NEXTVAL() to get the next value, CURRVAL() for current
- Sequences may have gaps - don't rely on consecutive numbers
- Independent sequences enable custom numbering schemes
