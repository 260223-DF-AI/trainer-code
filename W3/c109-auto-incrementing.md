# Auto-Increment

## Learning Objectives

- Understand auto-increment functionality
- Use SERIAL and BIGSERIAL types
- Work with sequences directly
- Manage identity columns effectively

## Why This Matters

Auto-incrementing primary keys are the most common way to generate unique identifiers in relational databases. Understanding how PostgreSQL implements this functionality helps you design tables correctly, troubleshoot gaps in sequences, and handle scenarios like bulk inserts or data migrations.

## The Concept

### What is Auto-Increment?

**Auto-increment** automatically generates the next integer value for a column, typically used for primary keys. Each new row gets a unique, incrementing number without manual specification.

```sql
CREATE TABLE users (
    user_id SERIAL PRIMARY KEY,  -- Auto-increments
    username VARCHAR(50)
);

INSERT INTO users (username) VALUES ('alice');  -- user_id = 1
INSERT INTO users (username) VALUES ('bob');    -- user_id = 2
INSERT INTO users (username) VALUES ('carol');  -- user_id = 3
```

### PostgreSQL SERIAL Types

PostgreSQL provides pseudo-types for auto-increment:

| Type | Storage | Range |
|------|---------|-------|
| SMALLSERIAL | 2 bytes | 1 to 32,767 |
| SERIAL | 4 bytes | 1 to 2,147,483,647 |
| BIGSERIAL | 8 bytes | 1 to 9,223,372,036,854,775,807 |

```sql
-- Small lookup table
CREATE TABLE status_codes (
    status_id SMALLSERIAL PRIMARY KEY,
    name VARCHAR(20)
);

-- Normal tables
CREATE TABLE products (
    product_id SERIAL PRIMARY KEY,
    name VARCHAR(100)
);

-- High-volume tables (events, logs)
CREATE TABLE events (
    event_id BIGSERIAL PRIMARY KEY,
    event_type VARCHAR(50),
    occurred_at TIMESTAMP
);
```

### How SERIAL Works

SERIAL is syntactic sugar that creates:

1. A sequence object
2. An integer column with DEFAULT nextval()

```sql
-- SERIAL is equivalent to:
CREATE SEQUENCE products_product_id_seq;
CREATE TABLE products (
    product_id INTEGER NOT NULL DEFAULT nextval('products_product_id_seq')
);
ALTER SEQUENCE products_product_id_seq OWNED BY products.product_id;
```

### IDENTITY Columns (SQL Standard)

PostgreSQL 10+ supports standard SQL IDENTITY columns:

```sql
-- GENERATED ALWAYS: database always generates value
CREATE TABLE orders (
    order_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    order_date DATE
);

-- GENERATED BY DEFAULT: use provided value or generate
CREATE TABLE logs (
    log_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    message TEXT
);
```

**GENERATED ALWAYS** vs **SERIAL**:

```sql
-- SERIAL allows override (can cause issues)
INSERT INTO products (product_id, name) VALUES (100, 'Override');  -- Works

-- GENERATED ALWAYS prevents override by default
INSERT INTO orders (order_id, order_date) VALUES (100, CURRENT_DATE);
-- ERROR: cannot insert into column "order_id"
-- Must use OVERRIDING SYSTEM VALUE if needed
INSERT INTO orders (order_id, order_date) OVERRIDING SYSTEM VALUE VALUES (100, CURRENT_DATE);
```

### Working with Sequences

**View current value**:

```sql
SELECT currval('products_product_id_seq');  -- After an insert
SELECT last_value FROM products_product_id_seq;  -- Direct query
```

**Get next value**:

```sql
SELECT nextval('products_product_id_seq');  -- Increments and returns
```

**Reset sequence**:

```sql
-- After bulk delete or truncate
ALTER SEQUENCE products_product_id_seq RESTART WITH 1;

-- Based on current max
SELECT setval('products_product_id_seq', (SELECT MAX(product_id) FROM products));
```

### Gaps in Auto-Increment

Gaps are normal and expected. They occur when:

- Transactions are rolled back
- Rows are deleted
- Inserts fail after sequence increment

```sql
BEGIN;
    INSERT INTO orders (customer_id) VALUES (1);  -- Gets order_id = 10
ROLLBACK;  -- Row not created, but sequence already incremented

INSERT INTO orders (customer_id) VALUES (2);  -- Gets order_id = 11
-- Gap: order_id 10 never exists
```

**Do not try to eliminate gaps.** They are harmless and trying to prevent them causes performance problems.

### After TRUNCATE or Bulk Delete

```sql
-- TRUNCATE resets the sequence (if RESTART IDENTITY)
TRUNCATE TABLE products RESTART IDENTITY;

-- After DELETE, sequence is not reset
DELETE FROM products;
-- Next insert still continues where sequence left off

-- Manually reset after delete
SELECT setval('products_product_id_seq', COALESCE((SELECT MAX(product_id) FROM products), 0));
```

### Bulk Insert Performance

For bulk inserts, pre-fetch sequence values:

```sql
-- Get next 1000 values
SELECT nextval('products_product_id_seq') FROM generate_series(1, 1000);

-- Or adjust sequence increment
ALTER SEQUENCE products_product_id_seq INCREMENT BY 1000;
SELECT nextval('products_product_id_seq');  -- Gets 1001
ALTER SEQUENCE products_product_id_seq INCREMENT BY 1;
```

## Code Example

Working with auto-increment:

```sql
-- Create table with SERIAL
CREATE TABLE customers (
    customer_id SERIAL PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE
);

-- Create table with IDENTITY
CREATE TABLE orders (
    order_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id INTEGER REFERENCES customers(customer_id),
    order_date DATE DEFAULT CURRENT_DATE
);

-- Insert data (auto-increment in action)
INSERT INTO customers (name, email) VALUES 
    ('Alice', 'alice@email.com'),
    ('Bob', 'bob@email.com'),
    ('Carol', 'carol@email.com');

-- Check generated IDs
SELECT * FROM customers;
-- customer_id: 1, 2, 3

-- Insert with RETURNING to get generated ID
INSERT INTO customers (name, email) 
VALUES ('David', 'david@email.com')
RETURNING customer_id;
-- Returns: 4

-- Create orders
INSERT INTO orders (customer_id) VALUES (1), (2), (1);

-- View sequence information
SELECT * FROM pg_sequences WHERE schemaname = 'public';

-- Check current sequence value
SELECT currval('customers_customer_id_seq');

-- Demonstrate gap creation
BEGIN;
    INSERT INTO customers (name) VALUES ('Test User');  -- Gets ID 5
ROLLBACK;

INSERT INTO customers (name, email) VALUES ('Eve', 'eve@email.com');
SELECT * FROM customers;
-- Eve gets ID 6 (5 is skipped due to rollback)

-- Reset sequence after cleanup
DELETE FROM customers WHERE customer_id > 4;
SELECT setval('customers_customer_id_seq', (SELECT MAX(customer_id) FROM customers));
-- Next insert will be ID 5

INSERT INTO customers (name, email) VALUES ('Eve', 'eve@email.com')
RETURNING customer_id;
-- Returns: 5
```

## Key Takeaways

- SERIAL/BIGSERIAL create auto-incrementing integer columns
- Use BIGSERIAL for high-volume tables
- IDENTITY columns are the SQL standard alternative
- Gaps in sequences are normal; do not try to eliminate them
- Use RETURNING to get generated values after INSERT
- Reset sequences after TRUNCATE if needed
